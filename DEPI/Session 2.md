### أول حاجة: الـ **Web Application** عبارة عن إيه؟
الموقع اللي بنفتحه في المتصفح بيتكون من جزئين رئيسيين:

1. **الكلاينت سايد** (اللي عندك في المتصفح)
   - HTML → الهيكل (العنوان، الفقرات، الصور...)
   - CSS → الشكل والألوان والتنسيق
   - JavaScript → الحركة والتفاعل (زر لما تضغط عليه يحصل حاجة)

2. **السيرفر سايد** (الجهاز اللي شغال فيه الموقع)
   - السيرفر بيستقبل طلبك
   - قاعدة بيانات (MySQL, MongoDB...) بتحفظ بيانات المستخدمين
   - كود بلغات زي Python, Node.js, PHP, Java بيعالج الطلب ويرد

الـ **URL** بتاع الموقع بيتكون من:
```
https://example.com/v1/users?search=google
```
- https → البروتوكول (آمن)
- example.com → الدومين الرئيسي
- /v1/users → المسار (Path) أو الـ Endpoint
- ?search=google → Parameters (اسم=القيمة)

### الثغرات اللي اتكلمنا عنها

#### 1. **XSS - Cross Site Scripting** (الأشهر والأخطر في البداية)
الثغرة دي بتحصل لما الموقع ياخد كلام منك (تعليق، اسم، بحث...) ويرجعه تاني من غير ما ينظفه، فأنت تقدر تحط **كود جافا سكريبت** جواه.

**أنواعه الثلاثة الرئيسية:**

| النوع              | وصف بسيط                                                                 | الخطورة          | مثال payload بسيط                  |
|---------------------|--------------------------------------------------------------------------|-------------------|-------------------------------------|
| Reflected XSS      | الكود بيجي في الـ URL أو في الطلب، ويظهر فورًا لك أنت بس             | متوسطة           | `<script>alert(1)</script>`        |
| Stored XSS         | الكود بيتخزن في الداتابيز (تعليق، بوست...)، أي حد يفتح الصفحة يتنفذ | عالية جدًا        | `<img src=x onerror=alert(1)>`     |
| DOM-based XSS      | المشكلة في الجافا سكريبت نفسه في المتصفح، مش بيعدي على السيرفر     | متوسطة → عالية   | `document.write(location.hash)`    |

**أهم حاجة تعمليها عشان تتأكدي إن في ثغرة:**
- جربي تحطي `"><script>alert(1337)</script>` أو `<img src=x onerror=alert(1337)>`
- لو طلع popup → تمام، في ثغرة
- دلوقتي كتير المواقع بتبلوك الـ `alert()`، جربي `print()` أو `confirm()` أو `prompt()`

**اللي ممكن يحصل لو حد استغلها:**
- يسرق الكوكيز (session cookie) → يدخل حسابك من غير باسوورد
- يعمل صفحة فيشينج داخل الموقع
- يغير شكل الصفحة (defacement)

**الحماية :**
- تنظيف الإنبوت (filter)
- تحويل الرموز الخطرة HTML entities (`<` → `&lt;`)
- استخدام **Content Security Policy** (CSP) header

#### 2. **Path Traversal / Directory Traversal** (اللي بيوصلك لملفات السيرفر)
الثغرة دي بتحصل لما الموقع بياخد اسم ملف منك (صورة، pdf...) وبيحطه في مسار على السيرفر من غير ما يتحقق.

مثال:
الموقع عايز يجيب صورة: `/images/abc.jpg`

لو كتبتي: `../../../../etc/passwd`  
→ السيرفر هيروح لفوق 4 مستويات ويجيب ملف المستخدمين في لينكس!

**Payloads شائعة:**
- `../../../etc/passwd`
- `..%2f..%2f..%2fetc%2fpasswd` (URL encoded)
- `....//....//etc/passwd` (bypass بعض الفلاتر)
- `../../../../windows/win.ini` (على ويندوز)

**الحماية:**
- ما تبعتيش اسم الملف من اليوزر أصلًا (استخدمي ID بدل الاسم)
- لو لازم → قائمة بيضاء (whitelist) + canonicalization (getCanonicalPath في جافا)

#### 3. **OS Command Injection** 
لما الموقع بياخد إنبوت منك وبيحطه في أمر نظام (command) زي `ping` أو `mail` أو `convert` من غير تنظيف.

مثال:
الموقع بيعمل: `ping -c 1 ` + اللي كتبته

لو كتبتي: `127.0.0.1; whoami`  
→ هيعمل ping وبعدين يقولك مين اليوزر اللي شغال بيه السيرفر!

**طرق كشفها:**
- `; whoami`
- `& dir` (Windoes)
- `| ls`
- Blind: `&& sleep 10` (لو الصفحة اتأخرت 10 ثواني → ثغرة)

**الحماية:**
- ما تستخدميش أوامر نظام أصلًا لو ممكن
- لو لازم → استخدمي API آمن (مش string concatenation)
- فلترة قوية + whitelist
